# Architecture Decision Record (ADR) 0001 — Multi-Tiered Self-Healing Platform

## Context & Problem Statement

Modern cloud-native operations on Kubernetes clusters suffer from slow manual incident response times, high developer alerts fatigue, and threat exposures during remediation executions. The system requires an autonomous incident detection, reasoning, validation, and execution engine that handles alerts safely.

## Decision Drivers

1. **Safety First**: Commands generated by AI must be parsed and blocked if they pose a threat to host filesystems.
2. **Reliability & Resiliency**: The system must run database and vector database searches cascadingly if dependencies fail (RTO < 1h, RPO < 5m).
3. **Durable Workflows**: Multi-stage agent reasoning loops must persist state checkpoints to avoid duplicate executions or lost tokens on crashes.
4. **WebSocket Scalability**: Streaming notifications to multiple client dashboards must scale horizontally.

## Considered Alternatives

1. **State Machine Frameworks**: Temporal (too heavy for local lightweight deployment), raw FastAPI routes (lacks durable resume capabilities), or custom DB-backed state transition tracking (selected).
2. **Vector DB Mappings**: Qdrant Server only (requires heavy network connections) vs Cascading Fallback Chain (Qdrant → Chroma → FAISS → In-Memory, selected for local dev resiliency).

## Decision Outcomes

- **Durable 8-State Workflow Machine**: Checkpoints are stored in the database at the end of each step. If a node restarts, the workflow recovers and resumes from the last uncompleted state.
- **Enkrypt AI Safety Envelope**: Hot-reloadable `policies.yaml` regex and command checks block executions and route tasks to manual validation gates.
- **Horizontal WebSocket pooling**: Managed using a clean pool with Redis Pub/Sub backplane logic and in-memory thread callback fallbacks.
